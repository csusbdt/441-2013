Overview

We meet twice per week. In addition to the weekly meetings, you can communicate with the class as a whole by posting to the CSUSB CSE 441 Spring 2013 Google Group. The course is heavily oriented towards programming, and you will need to learn and use C++ and Lua to complete projects in the course. If you are not sure you are ready for this course, then please discuss with me first.

The course will be oriented towards realtime 2D game programming with an emphasis on developing cross-platform code for desktop and mobile computers. Students only need to work in one development environment -- either Xcode, Visual Studio, or Linux -- but I will present on development in both Xcode and Visual Studio.

Throughout the course, we will develop a cross-platform approach to building highly performant games targeting Windows and OS X desktop environments. If someone in the class wants to pursue additional runtime environments -- such as Linux, iOS, Android, Blackberry, NaCl, etc -- then we will include these as well.

You will work in at least 2 languages in this course: C++ and Lua. Depending on the platform you want to focus on, you may need to work with another language, such as Objective-C for OS X or Java for Android. You will also need to learn at least the rudiments of low-level graphics programming.

We will use cocos2d-x as a reference project for study and experiments.

Course Goal

The goal of this course is to increase your understanding of video game programming, programming in general, and cross-platform development for mobile and desktop systems.

Learning Objectives

  • Learn how to use Git to manage source code.
  • Learn how to develop cross-platform video game projects based on C++, Lua, and OpenGL ES 2.0.

Assignments

In this course, you will primarily focus on research and experimentation resulting in a collection of small prototypes. You should strive to make your code as readable as possible.

Is source code well-organized? Have unnecessary variables and logic been removed from the code? Does the indentation and line spacing show logical structure? Is indentation and spacing done consistently? Are concerns separated when possible? Are the responsibilities of components clear and consistently assigned? Does the code contain names for variables, functions and classes that unambiguously express their purpose in the program? Are comments included when needed? Are superfluous comments omitted?

Program readability is important because real-world programs are read over and over again in the process of fixing bugs and adding new functionality.
